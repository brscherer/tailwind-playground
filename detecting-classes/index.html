Detecting Classes

How Tailwind finds & purges unused classes

How Class Detection Works

‚úÖ Static Classes (Detected)

These are found and included in the final CSS:

<div class="bg-blue-600 text-white"> <p class="mb-4 font-bold"> <button class="px-4 py-2 rounded">

‚ùå Dynamic Classes (Not Detected)

These are typically NOT found and will be purged:

<div class={'bg-' + color}> const cls = `text-${size}`; <div :class="`p-${value}`">

üí° Key Insight

Tailwind uses regex patterns to scan your source files for class names. It looks for strings that match Tailwind's class naming patterns. If a class isn't found in your source code, it won't be included in the final CSS bundle.

Content Configuration Patterns

1

Basic File Patterns

content: [
  "./index.html",
  "./src/**/*.{js,jsx,ts,tsx}",
]

Scans HTML and JavaScript files for class names. The most common pattern.

2

Framework Template Files

content: [
  "./src/**/*.vue",            // Vue
  "./src/**/*.svelte",       // Svelte
  "./app/**/*.{html,jsx,tsx}", // React
]

Include template files for your framework. Tailwind scans these for class names.

3

Safelist (Force Inclusion)

safelist: [
  'bg-red-500',
  'text-white',
  'p-4',
]

Manually list classes that won't be detected by content scanning. Use for dynamically generated classes.

4

Regex Patterns in Safelist

safelist: [
  { pattern: /bg-(red|green|blue)-(500|600)/ },
  { pattern: /text-/ },
  { pattern: /p-[0-9]/ },
]

Use regex patterns to match entire families of classes. More flexible than listing individual classes.

5

Blocklist (Exclude Classes)

blocklist: [
  'container',        // Problematic
  'px-0.5',           // Rarely used
]

Prevent certain classes from being generated even if found. Reduces bundle size.

Best Practices

‚úÖ Do's

  • ‚úì Use static class names - Hardcode classes when possible for full content scanning
  • ‚úì Configure content paths - Tell Tailwind where to find classes in your project
  • ‚úì Use safelist for dynamic classes - List dynamically generated classes explicitly
  • ‚úì Test production builds - Verify classes aren't being purged unexpectedly
  • ‚úì Use arbitrary values - These are detected and don't require safelist
  • ‚úì Keep class names whole - Never split class names with variables

‚ùå Don'ts

  • ‚úó Don't concatenate class strings - className = 'bg-' + color won't be detected
  • ‚úó Don't use ternary for partial classes - {isBig ? 'p-' : ''} problematic
  • ‚úó Don't forget to add content paths - Empty content array = no classes detected
  • ‚úó Don't ignore bundle warnings - If a class doesn't work, check detection
  • ‚úó Don't over-use safelist - It defeats the purpose of purging unused CSS
  • ‚úó Don't assume classes from comments - Tailwind won't detect them

Troubleshooting: Class Not Working?

Problem: "My class doesn't appear on page"

Likely cause: Class was purged because it wasn't detected.

Solutions:

  • Check that your file is included in the content array
  • Verify the class name is spelled correctly and matches Tailwind's format
  • If dynamically generated, add to safelist
  • Run build in dev mode to see if class appears

Problem: "My build is huge (lots of unused CSS)"

Likely cause: Content paths are too broad or safelist is overstuffed.

Solutions:

  • Be specific with content paths - don't include unnecessary directories
  • Review safelist and only include truly dynamic classes
  • Use regex patterns in safelist instead of listing every variation
  • Run pnpm build and check CSS file size

Problem: "Sometimes the class works, sometimes it doesn't"

Likely cause: Class is partially dynamic or depends on runtime conditions.

Solutions:

  • Make sure the complete class name appears in your source files
  • Don't split class names across variables/conditions
  • Use safelist with regex patterns to cover all possible values
  • Consider using inline styles for truly dynamic values

Advanced Configuration Example

// tailwind.config.js - Comprehensive example
export default {
  content: [
    "./index.html",
    "./src/**/*.{js,ts,jsx,tsx,vue}",
    "./components/**/*.{js,jsx,tsx}",
    "./pages/**/*.{js,jsx,tsx}",
  ],
  safelist: [
    // Explicitly included classes
    'h-px',
    'w-px',
    // Pattern-based inclusion
    {
      pattern: /bg-(red|green|blue|yellow)-(50|100|500|600)/,
    },
    {
      pattern: /text-(xs|sm|base|lg|xl)/,
    },
    // Variant-based inclusion
    {
      pattern: /^(hover|focus|active):/,
      variants: ['hover', 'focus', 'active'],
    },
  ],
  blocklist: [
    // Exclude specific classes to save space
    'container',
  ],
  theme: {
    extend: {},
  },
  plugins: [],
}

üìù Note: This configuration ensures Tailwind scans multiple directories, includes dynamic patterns, and uses blocklist to optimize bundle size.

Decision Tree: How to Handle Dynamic Classes

üìç START: You want to apply a class conditionally

‚ùì Is the class name complete and static in the source?

‚úÖ YES (class="p-4" or condition ? 'p-4' : 'p-2')

‚Üí It will be detected! No action needed.

‚ùì No - is it built from a small set of known values?

‚ö†Ô∏è YES (colors=['red','blue'], sizes=['sm','lg'])

‚Üí Add to safelist explicitly: safelist: ['bg-red-500', 'bg-blue-500', ...]

‚ùì No - is it a pattern (many possible values)?

‚ö†Ô∏è YES (any color, any size, any spacing)

‚Üí Use regex in safelist: pattern: /^bg-.*/ or use arbitrary values [#1e3a5f]

‚ùì No - truly arbitrary values?

‚ùå PROBABLY NOT (values from API, user input)

‚Üí Use inline styles instead: style={`color: ${userColor}`} or CSS variables
